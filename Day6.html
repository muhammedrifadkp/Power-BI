üî§ What is DAX?

Definition

DAX (Data Analysis Expressions) is a formula language used in Power BI, Power Pivot, and Analysis Services.

DAX allows you to:

  • Create custom calculations and aggregations
  • Build complex business logic
  • Perform time intelligence calculations
  • Create dynamic measures that respond to filters
  • Analyze data across multiple tables
  • Implement advanced statistical functions

DAX vs Excel Formulas

Aspect Excel Formulas DAX
Purpose Cell-by-cell calculations Column and table-level operations
References Cell references (A1, B2) Column references [ColumnName]
Syntax =SUM(A1:A10) =SUM(TableName[ColumnName])
Context Fixed cell position Dynamic filter context
Relationships Requires VLOOKUP/INDEX-MATCH Automatic via relationships
Performance Recalculates on every change Optimized for large datasets
Learning Curve Easy to start Steeper but more powerful
üìù Good News: If you know Excel formulas, you already have a head start! Many DAX functions have similar names (SUM, AVERAGE, IF, etc.)

Where DAX is Used

1. Calculated Columns

Add new columns to tables with custom calculations

Example: Profit = Revenue - Cost

2. Measures

Create dynamic aggregations that respond to filters

Example: Total Sales = SUM([Revenue])

3. Calculated Tables

Generate entire new tables from DAX expressions

Example: Date table, What-if parameters

4. Row-Level Security

Define security rules using DAX filters

Example: [Region] = USERNAME()

Understanding Evaluation Context

Evaluation Context: The environment in which a DAX expression is evaluated. This is what makes DAX powerful but also tricky!

Two Types of Context:

1. Row Context

When: Calculated columns

What: DAX evaluates formula row by row

Example: In each row, [Profit] = [Revenue] - [Cost]

2. Filter Context

When: Measures

What: DAX respects current filters from slicers, visuals, row/column headers

Example: SUM(Sales[Revenue]) changes based on selected product, date, etc.

üéØ Key Insight: Understanding context is THE most important concept in DAX. We'll see this in action throughout today's lab!

‚öñÔ∏è Calculated Columns vs Measures

The Big Difference

üìä Calculated Columns

What: Add a new column to a table

When Calculated: At data refresh time

Storage: Stored in the model (uses memory)

Context: Row context

Icon: Calculator icon next to column name

üìê Measures

What: Dynamic calculation formula

When Calculated: At query time (when visual renders)

Storage: Only formula stored (no data)

Context: Filter context

Icon: Calculator with fx symbol

Detailed Comparison

Feature Calculated Column Measure
Creation Modeling ‚Üí New Column Modeling ‚Üí New Measure
Syntax Start Column Name = Measure Name :=
Result Value in every row Single aggregated value
Can Reference Other columns in same row Other measures, aggregations
Can Use In Slicers, rows/columns, filters Values area of visuals
Performance Slower (stored data) Faster (calculated on-demand)
File Size Increases model size No impact on size
Use For Static values, categorization Aggregations, KPIs, calculations

Calculated Column Examples

Example 1: Simple Math

Profit = Sales[Revenue] - Sales[Cost]

Result: Every row gets a Profit value

Row 1: $1000 - $600 = $400

Row 2: $500 - $300 = $200

Example 2: Conditional Logic

Sales Category = IF(Sales[Revenue] >= 1000, "High", "Low")

Result: Each sale is categorized as High or Low

Example 3: Text Concatenation

Full Name = Customers[FirstName] & " " & Customers[LastName]

Result: Combines first and last names

Measure Examples

Example 1: Simple Aggregation

Total Revenue := SUM(Sales[Revenue])

Result: Sums all revenue based on current filters

Changes dynamically based on slicers!

Example 2: Average

Average Sale := AVERAGE(Sales[Revenue])

Result: Average of visible sales

Example 3: Calculated Measure

Profit Margin := DIVIDE([Total Profit], [Total Revenue])

Result: Calculates margin using other measures

When to Use Each

‚úÖ Use Calculated Column When:

  • You need to filter or slice by the result
  • You need row-by-row calculations
  • Creating categories or segments
  • Combining text fields
  • Creating bins or ranges
  • Result doesn't change with filters

Examples: Full Name, Age Group, Profit (Revenue - Cost)

‚úÖ Use Measure When:

  • You need aggregations (SUM, AVERAGE, COUNT)
  • Calculation should respond to filters
  • Creating KPIs and metrics
  • Building ratios or percentages
  • Time intelligence calculations
  • You want optimal performance

Examples: Total Sales, Average Price, YTD Revenue, Profit %

üéØ Golden Rule: When in doubt, use a MEASURE! Measures are more flexible, faster, and don't increase file size.

Implicit vs Explicit Measures

Implicit Measures (Auto-created)

What: Power BI automatically creates when you drag a numeric column to Values

Example: Drag [Revenue] to Values ‚Üí Creates "Sum of Revenue"

Problem: Limited control, can't reference in other measures, not reusable

Explicit Measures (You create)

What: Measures you deliberately create with specific formulas

Example: Create measure "Total Revenue := SUM(Sales[Revenue])"

Benefits: Full control, can reference in other calculations, reusable, documented

üí° Best Practice: Always create explicit measures! Don't rely on implicit measures for important calculations.

üìù Understanding DAX Syntax & Operators

Basic DAX Syntax Rules

1. Naming Conventions

  • Table names: 'Sales', Orders (use single quotes if spaces)
  • Column names: [Revenue], [Product Name] (square brackets)
  • Measures: [Total Sales] (also square brackets in formulas)
  • Full reference: Sales[Revenue] (Table[Column])

2. Creating Formulas

  • Calculated Column: New Column Name = formula
  • Measure: Measure Name := formula
  • Note: := is preferred for measures, but = also works

3. Case Sensitivity

  • DAX is NOT case-sensitive
  • SUM, Sum, sum all work the same
  • But use consistent casing for readability!

DAX Operators

Arithmetic Operators

+ Addition [Price] + [Tax]
- Subtraction [Revenue] - [Cost]
* Multiplication [Quantity] * [Price]
/ Division [Total] / [Count]
^ Exponentiation [Value] ^ 2

Comparison Operators

= Equal to [Status] = "Active"
<> Not equal [Region] <> "South"
> Greater than [Sales] > 1000
< Less than [Age] < 30
>= Greater/equal [Score] >= 90
<= Less/equal [Price] <= 100

Logical Operators

&& AND [Sales] > 100 && [Region] = "West"
|| OR [Status] = "A" || [Status] = "B"
NOT NOT NOT([Status] = "Inactive")

Text Operator

& Concatenate [First] & " " & [Last]

Syntax Examples

‚úÖ Correct Syntax Examples

Total Sales := SUM(Sales[Revenue])

‚úì Measure with clear name and table reference

Profit Margin := DIVIDE([Total Profit], [Total Revenue], 0)

‚úì Uses measures in formula with DIVIDE for safe division

Full Name = [First Name] & " " & [Last Name]

‚úì Calculated column concatenating text

Sales Rank = IF([Revenue] >= 1000, "High", IF([Revenue] >= 500, "Medium", "Low"))

‚úì Nested IF for multiple conditions

‚ùå Common Syntax Errors

Total Sales = SUM(Revenue)

‚úó Missing table name - should be Sales[Revenue]

Profit = Revenue - Cost

‚úó Missing brackets - should be [Revenue] - [Cost]

Average := Sales[Revenue] / COUNT()

‚úó COUNT needs a column - use AVERAGE() instead

Total = SUM(Sales[Revenue]

‚úó Missing closing parenthesis

üî¢ Basic DAX Functions

Aggregation Functions

SUM

Purpose: Add up all values

Total Sales := SUM(Sales[Revenue])

Use when: Totaling numbers (revenue, quantity, cost)

AVERAGE

Purpose: Calculate mean value

Avg Price := AVERAGE(Products[Price])

Use when: Finding typical value (average sale, price)

COUNT

Purpose: Count numeric values

Transaction Count := COUNT(Sales[SalesID])

Use when: Counting transactions, occurrences

COUNTA

Purpose: Count non-blank values

Customer Count := COUNTA(Sales[CustomerID])

Use when: Counting text or mixed data

DISTINCTCOUNT

Purpose: Count uniquevalues

Unique Customers := DISTINCTCOUNT(Sales[CustomerID])

Use when: Counting unique items (customers, products)

MIN / MAX

Purpose: Find smallest/largest value

Max Sale := MAX(Sales[Revenue]) Min Sale := MIN(Sales[Revenue])

Use when: Finding extremes

Mathematical Functions

DIVIDE

Purpose: Safe division (handles divide by zero)

Profit Margin := DIVIDE([Total Profit], [Total Revenue], 0)

Syntax: DIVIDE(numerator, denominator, alternate_result)

Third parameter is what to return if division by zero

ROUND

Purpose: Round to specified decimals

Rounded Price = ROUND([Price], 2)

Use when: Cleaning decimal places

ABS

Purpose: Absolute value (remove negative)

Abs Variance = ABS([Actual] - [Budget])

Use when: Variance analysis

Logical Functions

IF

Purpose: Conditional logic

Status = IF([Revenue] > 1000, "High", "Low")

Syntax: IF(condition, value_if_true, value_if_false)

SWITCH

Purpose: Multiple conditions (cleaner than nested IF)

Region Name = SWITCH([RegionID], 1, "North", 2, "South", 3, "East", "West")

Use when: Many conditions to check

AND / OR

Purpose: Combine conditions

Premium = IF(AND([Revenue] > 5000, [Customer Type] = "Gold"), "Yes", "No")

Use when: Multiple criteria needed

Step-by-Step: Creating a Measure

üìù

How Power BI Calculates a Measure

Example Measure: Total Sales := SUM(Sales[Revenue])

What Happens When You Use This Measure:

  1. User interacts with report (e.g., selects "2024" in year slicer)
  2. Power BI determines filter context
    • Current filters: Year = 2024
    • Visual context: Maybe also Product = "Laptop"
  3. DAX engine applies filters
    • Filters Sales table to matching rows
    • Only rows where Year = 2024 AND Product = "Laptop"
  4. SUM function executes
    • Adds up [Revenue] for filtered rows
    • Returns single number
  5. Result displays in visual

This is why measures are dynamic! Same formula, different results based on filters.

DAX Best Practices

‚úÖ DO's

  • Use clear, descriptive names
  • Always use table references: Table[Column]
  • Use DIVIDE instead of / for division
  • Format measures properly (currency, percentage)
  • Add descriptions to complex measures
  • Organize measures in display folders
  • Test measures with different filters
  • Use variables for repeated calculations

‚ùå DON'Ts

  • Don't use calculated columns for aggregations
  • Don't create circular references
  • Don't use column references without table name
  • Don't ignore errors or warnings
  • Don't create overly complex formulas
  • Don't forget to test edge cases
  • Don't use vague names like "Calc1", "Measure2"

1

Add Calculated Columns to Sales Table

  1. Open your Day 5 Power BI file
  2. Go to Data View
  3. Select the Sales table
  4. Click Table tools ‚Üí New column
  5. Column 1: Total Amount
    • Formula: Total Amount = Sales[Quantity] * Sales[UnitPrice]
    • Press Enter
    • Verify: Each row should show quantity √ó price
  6. Column 2: Sales Category
    • Formula: Sales Category = IF(Sales[Total Amount] >= 1000, "High", IF(Sales[Total Amount] >= 500, "Medium", "Low"))

Exercise 2: Create Basic Measures (30 minutes)

2

Build Essential Measures

  1. Stay in Data View or switch to Report View
  2. Select Sales table in Fields pane
  3. Click Table tools ‚Üí New measure
  4. Measure 1: Total Revenue
    Total Revenue := SUM(Sales[Total Amount])
    • Press Enter
    • Format as Currency: Select measure ‚Üí Measure tools ‚Üí Format: Currency ‚Üí $ English (United States)
  5. Measure 2: Total Quantity
    Total Quantity := SUM(Sales[Quantity])
    • Format as Whole Number
  6. Measure 3: Average Sale
    Average Sale := AVERAGE(Sales[Total Amount])
    • Format as Currency
  7. Measure 4: Number of Transactions
    Transaction Count := COUNT(Sales[SalesID])
    • Format as Whole Number
  8. Measure 5: Unique Customers
    Unique Customers := DISTINCTCOUNT(Sales[CustomerKey])
    • Format as Whole Number

Exercise 4: Create Advanced Measures (15 minutes)

4

Build Measures Using Other Measures

  1. Measure: Average per Customer
    Avg per Customer := DIVIDE([Total Revenue], [Unique Customers], 0)
    • Format as Currency
    • This shows average revenue per unique customer
  2. Measure: Average Items per Transaction
